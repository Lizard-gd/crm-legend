## Доступ к базе данных

В проекте используется общий PostgreSQL дев-стенда с изоляцией по схемам.

- **Приложение** подключается к базе напрямую по JDBC — оно читает/пишет данные и применяет миграции Liquibase.
- **Человек** (разработчик) работает с базой только через pgAdmin — это единственная разрешённая дверь для диагностики и просмотра данных.
- Все учётные данные (логины, пароли, хосты) хранятся в **Vault** и выдаются через онбординг; в репозиторий и код они не попадают.

PgAdmin используется для наблюдения и read-only запросов. Создание и изменение структуры таблиц выполняется только через миграции Liquibase.

## Моя история:
У Банка не было единой системы для корпоративных клиентов – данные хранились в файлах и таблицах, из-за этого они терялись и реальную воронку продаж было сложно отследить, от этого появлялась упущенная прибыль. Банку была нужна независимая CRM, которую можно развивать и поддерживать в долгосрок. Я работал над такой системой: она должна была централизовать хранение клиентов, делать работу менеджеров проще, понятней и давать чёткую картину этапов сделки. Я разрабатываю бэкенд – отвечаю за бизнес-логику и безопасность данных. Работал по принципу Contract‑First: в компании уже была описана API в спецификации OpenAPI (Swagger), по ней я реализовал и внедрял новые фичи и тестировал, в том числе, с помощью H2 DataBase. В спринтах я добавляю функциональность по управлению заявками, валидацию, интеграцию с партнёрами и аудит операций. Стек: Spring Boot, PostgreSQL, Liquibase, JPA, OpenAPI, контейнеризация в Kubernetes.
